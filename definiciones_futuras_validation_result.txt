## Definiciones futuras a tener en cuenta para `validation_result.py`

Estas definiciones no deben implementarse todavía, pero conviene dejarlas establecidas para orientar la evolución del modelo.

# 1. Separación entre error crítico y observación no bloqueante

## Futuro criterio
El modelo deberá distinguir al menos tres niveles:

- Error crítico
- Advertencia
- Observación informativa

## Implicancia
No todo hallazgo debe invalidar el documento.

### Ejemplos
Errores críticos
- CAE ausente
- total ausente
- emisor CUIT ausente
- fecha comprobante ausente
- estructura de comprobante inválida

Advertencias
- producto/servicio incompleto
- receptor no detectado con exactitud, pero parecido a una variante válida
- formato de fecha ambiguo
- percepciones no identificadas

Observaciones
- moneda inferida
- descripción resumida
- OCR utilizado como fallback

# 2. Diferenciar validez automática de validez operativa

## Futuro criterio
Debe existir diferencia entre:

- is_valid_automatic
- is_valid_for_manual_review
- is_approved_final

## Motivo
Un documento puede:
- no ser validable automáticamente
- pero sí ser apto para revisión humana
- y luego ser aprobado manualmente

## Implicancia
is_valid como único booleano será insuficiente más adelante.

# 3. Clasificación de faltantes por criticidad

## Futuro criterio
missing_fields no debería ser una sola lista plana.

Debería separarse en:
- missing_critical_fields
- missing_secondary_fields

## Ejemplo
Críticos
- tipo comprobante
- punto de venta
- número
- fecha
- emisor cuit
- CAE
- total

Secundarios
- producto_servicio
- fecha_vencimiento_cae
- moneda inferida
- detalle de alícuotas

# 4. Validación por reglas de negocio parametrizables

## Futuro criterio
El resultado de validación no debería depender solo de lógica fija en código.

Debe poder reflejar reglas externas como:
- tolerancia de redondeo
- receptor permitido
- monedas válidas
- tipos documentales admitidos
- obligatoriedad de ciertos campos

## Implicancia
ValidationResult deberá ser compatible con una validación configurable por:
- archivo .env
- JSON/YAML
- hoja PARAMETROS
- base de configuración local

# 5. Trazabilidad de qué regla falló

## Futuro criterio
No alcanza con guardar solo el mensaje textual.

Conviene registrar:
- regla ejecutada
- resultado
- severidad
- campo afectado

## Posible estructura futura
Cada validación podría generar objetos tipo:

- rule_code
- field_name
- severity
- message
- passed

## Motivo
Esto permitirá:
- debugging
- auditoría
- UI más clara
- explicación del porqué de una invalidación

# 6. Compatibilidad con revisión manual en UI

## Futuro criterio
El modelo debe prepararse para que un usuario pueda intervenir sobre el resultado.

Por eso más adelante podría incluir:
- review_required
- reviewed_by
- reviewed_at
- manual_override
- override_reason

## Motivo
El flujo real del sistema no será 100 % automático.

# 7. Manejo específico de duplicados

## Futuro criterio
duplicate_detected no debería necesariamente invalidar el documento.

Debe distinguirse entre:
- duplicado exacto confirmado
- posible duplicado
- coincidencia parcial
- duplicado aprobado manualmente

## Posibles campos futuros
- duplicate_status
- duplicate_match_type
- duplicate_candidates
- duplicate_review_required

# 8. Nivel de confianza más estructurado

## Futuro criterio
confidence_level no debería depender solo de un texto libre como:
- Alta
- Media
- Baja

Más adelante conviene agregar:
- confidence_score numérico
- confidence_factors

## Ejemplo
- score: 82.5
- factores:
  - fecha detectada
  - total detectado
  - CAE detectado
  - receptor validado
  - importes consistentes

## Motivo
Esto mejora:
- ranking de calidad
- priorización de revisión
- métricas
- UI

# 9. Compatibilidad con múltiples tipos documentales

## Futuro criterio
Aunque hoy el foco es factura, ValidationResult debería poder evolucionar para:
- remitos
- notas de crédito
- certificados
- otros comprobantes

## Implicancia
El modelo no debería quedar conceptualmente atado solo a una factura, aunque hoy se use en ese contexto.

# 10. Estado final de validación

## Futuro criterio
Más adelante conviene representar explícitamente estados como:

- pending_validation
- validated_automatic
- requires_manual_review
- validated_manual
- rejected

## Motivo
Un booleano is_valid no alcanza para representar el ciclo real del documento.

# 11. Registro de inconsistencias numéricas más detallado

## Futuro criterio
No solo guardar rounding_difference.

También podría registrarse:
- subtotal esperado
- iva esperado
- percepciones esperadas
- total esperado
- total detectado
- diferencia exacta

## Motivo
Permite explicar claramente por qué falló la validación de importes.

# 12. Preparación para exportación y reporting

## Futuro criterio
El modelo debe seguir siendo fácilmente serializable.

Por eso conviene mantener compatibilidad futura con:
- to_dict()
- to_json()
- to_row_dict()
- exportación tabular

# Síntesis operativa

## En esta etapa
validation_result.py debe seguir siendo:
- simple
- claro
- funcional

## Pero debe evolucionar luego hacia un modelo con:
- severidades
- reglas identificables
- soporte de revisión manual
- estados de validación
- score de confianza
- duplicados más ricos
- validación configurable

## Recomendación
Conviene dejar estas definiciones como criterio de diseño, pero no implementarlas todavía para no sobredimensionar el MVP.
